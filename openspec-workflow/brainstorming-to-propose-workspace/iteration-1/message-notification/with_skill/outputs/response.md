# 测试用例 3：站内消息通知模块（有 skill）

## 用户输入

> 我想给系统加个站内消息通知模块，用户登录后能看到未读消息，点击消息可以跳转到对应业务页面。先帮我想想怎么做比较好。

---

## 阶段 1：brainstorming 需求探索

这个需求涉及的技术选型比较多，我先确认几个核心决策点：

### 澄清问题

**1. 消息实时性要求**

- A. 实时推送（WebSocket）— 消息产生后立即弹出通知
- B. 短轮询（每 N 秒请求一次）— 延迟几秒但实现简单
- C. 长轮询（SSE 或 comet）— 折中方案

**2. 消息展示方式**

- A. 顶部导航栏铃铛图标 + 下拉消息列表（类似 GitHub/GitLab）
- B. 独立的消息中心页面
- C. 两者都要（铃铛看摘要 + 点击进入消息中心看详情）

**3. 消息类型与跳转**

你提到"跳转到对应业务页面"，消息的类型有哪些？

- A. 系统通知（公告、维护通知等）
- B. 业务消息（审批结果、工单更新等）
- C. 以上都有

消息跳转的粒度：
- A. 跳转到具体页面即可
- B. 需要跳转到页面并定位到具体数据行（如某个审批单）

**4. 消息管理能力**

- A. 仅查看和标记已读
- B. 需要删除/归档
- C. 需要按类型筛选

**5. 未读消息的提醒方式**

- A. 铃铛图标上显示未读数量 badge
- B. 首次打开时弹出未读提醒
- C. 浏览器原生通知（Notification API）

---

### 假设用户回复：B、C、C+A、A、A

### 方案比较

| 方案 | 实时性 | 复杂度 | 推荐度 |
|------|--------|--------|--------|
| WebSocket | 实时 | 高（需后端配合、断线重连、心跳） | 适合消息量大、实时要求高的场景 |
| 短轮询（推荐） | 延迟 10-30s | 低（纯 HTTP 请求） | 适合当前阶段，后续可升级 |
| SSE | 准实时 | 中 | 需后端支持，单向推送够用 |

**推荐方案：短轮询**

理由：
- 当前是 MVP 阶段，消息量不大
- 不需要后端额外搭建 WebSocket 服务
- 短轮询 + 浏览器缓存可以满足 10-30s 延迟
- 后续如果实时性要求提高，可以平滑升级到 WebSocket

---

### 硬确认门

> 需求探索阶段完成，以下是结论摘要：
>
> - **目标**：为系统新增站内消息通知模块，支持未读消息查看、消息跳转、已读标记
> - **范围**：做 — 铃铛 badge + 下拉消息列表 + 消息中心页面 + 短轮询获取 + 已读标记 + 消息跳转；不做 — WebSocket 推送、消息删除归档、浏览器原生通知、消息筛选
> - **推荐方案**：短轮询（10-30s 间隔）+ 顶部铃铛 + 消息中心页，后续可升级 WebSocket
> - **核心约束**：当前为 MVP 阶段，后端消息接口待确认
>
> 确认后将进入 `/opsx:propose` 创建正式提案。是否继续？

### 阶段 2：结构化摘要（假设用户已确认）

**change-name：** `message-notification-module`

**proposal 内容：**
- Why：用户需要及时获知系统通知和业务消息（审批结果、工单更新等），目前缺少统一的消息触达渠道
- What：新增站内消息通知模块，包含顶部铃铛入口、下拉消息列表、独立消息中心页面
- Scope：前端消息展示层 + 短轮询机制，后端消息接口需配合开发
- Impact：所有登录用户可查看未读消息并快速跳转到对应业务页面

**spec 内容：**
- 顶部导航栏铃铛图标显示未读消息数量 badge
- 点击铃铛展开下拉面板，展示最近 10 条未读消息
- 下拉面板底部有"查看全部"链接，进入消息中心页面
- 消息中心页面按时间倒序展示所有消息（已读 + 未读）
- 未读消息有视觉区分（加粗/背景色）
- 点击消息后标记为已读并跳转到对应业务页面
- 系统公告类型消息无跳转，点击仅标记已读
- 业务消息跳转时携带业务 ID，目标页面定位到具体数据
- 短轮询间隔 15 秒，页面不可见时暂停轮询
- 用户退出登录时停止轮询

**design 内容：**
- 技术选型：Vue 3 Composition API + Pinia + TypeScript
- 状态管理：`useMessageStore`（Pinia），管理未读数、消息列表、轮询状态
- 组件拆分：
  - `MessageBell` — 顶部铃铛 + badge
  - `MessageDropdown` — 下拉消息面板
  - `MessageCenter` — 独立消息中心页面（`/messages`）
- 轮询机制：`useIntervalFn` + `document.visibilitychange` 控制暂停
- 接口设计：
  - `GET /api/messages/unread-count` — 未读数量
  - `GET /api/messages?page=1&size=20` — 消息列表
  - `PATCH /api/messages/:id/read` — 标记已读
  - `PATCH /api/messages/read-all` — 全部已读

**tasks 雏形：**
1. 定义消息类型接口和 store
2. 实现 useMessageStore（状态管理 + 轮询逻辑）
3. 实现 MessageBell 组件（铃铛 + badge）
4. 实现 MessageDropdown 组件（下拉消息面板）
5. 实现 MessageCenter 页面（消息中心）
6. 实现消息跳转逻辑（路由 + 业务 ID 定位）
7. 添加路由配置（/messages）
8. 在顶部导航栏集成 MessageBell
9. 联调测试
